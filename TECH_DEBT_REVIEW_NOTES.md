# 포트폴리오 대시보드 코드리뷰 (기술부채 관점)

## 1) 현재 구현 기능 요약 (간단 노트)

- 로그인/인증
  - Streamlit 앱 진입 시 AWS SSM Parameter Store에서 대시보드 비밀번호를 조회하고 세션 상태로 인증을 유지함.
- 데이터 수집 파이프라인
  - 한국투자증권(개인/법인), 키움증권(법인) API를 호출해 주식/예수금 정보를 통합 수집함.
  - 키움 API는 환경변수로 스킵 가능.
- 환율 처리
  - 외부 환율 API(open.er-api.com)에서 환율을 받아 통화별 평가금액/손익을 원화로 환산함.
  - 실패 시 하드코딩된 기본 환율 사용.
- 포트폴리오 현황 탭
  - 계좌 필터, 핵심 KPI(총평가/원금/손익/수익률/예수금), 계좌별/종목별/국내-해외 비중 파이차트 제공.
  - 차트 클릭 이벤트로 국내/해외 상세 종목 드릴다운.
  - 계좌별 상세 보유표, 전체 종목 요약표, CSV 다운로드, 예수금 상세 제공.
- 성과 분석 탭
  - 현재는 플레이스홀더와 기본 통계만 제공(고도화 예정 기능 목록 포함).
- 토큰/설정 관리
  - 브로커 API 토큰을 파일 캐시로 저장/재사용.
  - 계정 키/시크릿/계좌번호는 AWS SSM에서 로드.

## 2) 기술부채 리뷰

### A. 우선순위 높음 (안정성/운영 리스크)

1. **대시보드 렌더링 코드의 단일 파일 집중(거대한 `dashboard_app.py`)**
   - UI, 데이터 가공, 차트 로직, 세션 이벤트 처리, 스타일 정의가 한 파일에 밀집됨.
   - 차트 섹션(특히 국내/해외 비중 + 드릴다운 + 상세표)이 길고 중복이 많아 변경 시 회귀 위험이 큼.
   - 개선: `ui/`, `services/`, `transform/`, `charts/` 모듈로 분리하고, 차트 생성 함수를 순수 함수로 분리.

2. **데이터 변환에서 `DataFrame.apply(axis=1)` 다중 사용으로 성능/가독성 저하**
   - 원화 환산/원금 계산/국가 계산이 행 단위 apply에 의존.
   - 데이터가 늘어나면 성능 병목 가능.
   - 개선: 벡터화 연산(`map`, `where`, `np.select`)으로 교체하고 스키마 검증(필수 컬럼 체크) 추가.

3. **예외 처리의 과도한 broad catch (`except Exception`)**
   - AWS/외부 API 실패 시 원인 분류가 어렵고 복구 전략이 없음.
   - 개선: 네트워크 예외/인증 예외/파싱 예외를 분리하고 사용자 메시지 vs 운영 로그를 분리.

4. **민감정보/토큰 파일 저장 방식의 운영 보안 부채**
   - 토큰이 로컬 파일(`tokens/`)에 저장되어 실행 환경에 따라 권한/정리 정책이 불명확.
   - 개선: 파일 퍼미션 제한, 경로 외부화, 만료/삭제 정책, 또는 Redis/Secrets Manager 등 중앙 저장소 사용 검토.

5. **브로커 연동 코드의 추상화 미흡**
   - KIS/Kiwoom 파싱 로직은 있으나 공통 인터페이스(예: DTO, 표준 에러 타입) 정립이 약함.
   - 개선: `BrokerClient` 프로토콜 + 표준 `AssetRecord` dataclass/Pydantic 모델 도입.

### B. 우선순위 중간 (유지보수성/확장성)

6. **하드코딩 값 다수 (색상, 문자열, 기본 환율, 파라미터 키 패턴)**
   - 변경 시 코드 수정 범위가 넓고 실수 가능성이 큼.
   - 개선: `config.py`/`constants.py`로 분리하고 환경/배포별 설정 계층화.

7. **도메인 규칙의 코드 분산**
   - 계좌 라벨 규칙, 시장 구분, 통화 처리, 표기 규칙이 여러 위치에 흩어져 중복.
   - 개선: 도메인 매핑 딕셔너리/헬퍼로 일원화.

8. **로깅 전략 부재 (`print`, `st.error` 혼용)**
   - 운영에서 검색/추적/레벨 제어가 어려움.
   - 개선: 표준 `logging` 도입, 구조화 로그(JSON) + correlation id + 사용자 노출 메시지 분리.

9. **타입 안정성 부족**
   - `dict` 기반 loose schema 사용으로 런타임 오류 가능성 존재.
   - 개선: dataclass/Pydantic 모델 + 정적 타입 검사(`mypy`) 도입.

10. **테스트 부재**
    - 계산 로직(원금/손익/비중), 브로커 파서, 환율 fallback 등에 대한 자동 테스트가 없음.
    - 개선: 단위테스트(pytest) 우선 도입 후 API mocking으로 회귀 테스트 구성.

### C. 빠른 개선 Quick Wins

- `dashboard_app.py` 내 반복되는 색상 맵 생성/차트 레이아웃 설정을 함수로 추출.
- 환율 fallback 값을 상수화하고 업데이트 기준일 메타데이터를 함께 표기.
- 숫자 컬럼 정규화 함수를 만들어 모든 테이블 생성 전에 공통 적용.
- 오류 메시지 템플릿 통일(사용자용/운영자용).

## 3) 기술부채 제거 로드맵 (제안)

1. **1주차: 안전망 구축**
   - 핵심 계산 로직 단위테스트 추가(수익률, 원화 환산, 집계).
   - 로깅 표준화(`logging`) 및 에러 코드 도입.

2. **2주차: 구조 분리**
   - `dashboard_app.py`를 `pages/components/services`로 분해.
   - 브로커 출력 포맷을 `AssetRecord` 모델로 표준화.

3. **3주차: 성능/품질 개선**
   - `apply(axis=1)` 벡터화 전환.
   - 린트/타입체크(`ruff`, `mypy`) + CI 파이프라인 연동.

4. **4주차: 운영성 강화**
   - 토큰 저장 정책 개선(권한, 만료, 삭제).
   - 관측성(실패율, API latency, 환율 API 실패 fallback 카운트) 지표 추가.
